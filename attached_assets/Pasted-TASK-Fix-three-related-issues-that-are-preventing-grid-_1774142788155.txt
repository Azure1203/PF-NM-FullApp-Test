TASK: Fix three related issues that are preventing grid row lookups 
from working in both the formula tester and the order pipeline.

═══════════════════════════════════════════════════
FIX 1 — Grid import keyColumn priority (critical)
═══════════════════════════════════════════════════

BACKGROUND:
In server/routes.ts, the grid CSV upload handler determines keyColumn 
in this order:
  1. If headers include "EXISTING OPTION ID" → keyColumn = that
  2. Else if headers include "MANU_CODE" → keyColumn = MANU_CODE
  3. Else → NAME

This is wrong. "EXISTING OPTION ID" is a database ID with no meaning 
to the pricing pipeline. The correct priority is:
  1. MANU_CODE (most specific — directly matches order CSV SKUs)
  2. NAME (human-readable code like TFL1W — matches color CSV values)
  3. EXISTING OPTION ID (last resort only)

EVERY grid that has been imported already has the wrong keyColumn and 
wrong lookupKey values stored. The rows need to be re-imported once 
the priority is fixed.

FIX in server/routes.ts — the upload handler for single grids 
(POST /api/admin/upload-dynamic-grid) and the batch upload handler 
if one exists:

Change the keyColumn detection from:
  if (headers.includes('EXISTING OPTION ID')) {
    keyColumn = 'EXISTING OPTION ID';
  } else if (headers.includes('MANU_CODE')) {
    keyColumn = 'MANU_CODE';
  }

To:
  if (headers.includes('MANU_CODE')) {
    keyColumn = 'MANU_CODE';
  } else if (headers.includes('NAME')) {
    keyColumn = 'NAME';
  } else if (headers.includes('EXISTING OPTION ID')) {
    keyColumn = 'EXISTING OPTION ID';
  }

After this fix is deployed, every grid must be re-imported by 
uploading its CSV again through the Attribute Grid Manager. The 
admin will need to do this manually for each grid — this is a 
one-time operation after the code is deployed.

═══════════════════════════════════════════════════
FIX 2 — rowDataColumn fallback uses wrong column
═══════════════════════════════════════════════════

BACKGROUND:
The three-step grid row lookup (exact → case-insensitive → rowData 
column fallback) currently passes binding.lookupColumn as the 
rowDataColumn fallback. For the color binding, lookupColumn = 
"Material" (the order CSV column name) — but "Material" doesn't 
exist in the color grid's rowData. This makes the fallback useless.

The fallback should search ALL rowData columns for the value, not 
just one specific column. This makes the lookup resilient regardless 
of how the grid was imported or what lookupColumn is set to.

FIX in server/storage.ts — getAttributeGridRowByKey:
Replace the current third step (which only searches one column) with 
a search across all rowData values:

  // Step 3: search ALL rowData values for a match
  if (rowDataColumn) {
    const byColumn = allRows.find(r => {
      const rd = r.rowData as Record<string, any>;
      const val = rd[rowDataColumn] 
        ?? rd[rowDataColumn.toLowerCase()] 
        ?? rd[rowDataColumn.toUpperCase()];
      return String(val ?? '').trim().toLowerCase() 
        === trimmedKey.toLowerCase();
    });
    if (byColumn) return byColumn;
  }
  // Step 4 (new): search ALL rowData values as a last resort
  return allRows.find(r => {
    const rd = r.rowData as Record<string, any>;
    return Object.values(rd).some(v => 
      String(v ?? '').trim().toLowerCase() === trimmedKey.toLowerCase()
    );
  });

Apply the same change to findGridRowInCache in server/routes.ts 
(the in-memory equivalent used in the upload pipeline and reprice 
route) — add the same "search all rowData values" step after the 
existing three steps.
